# Sotuvlar (Продажи / Orders) — расхождения дизайна и API

Документ-памятка по модулю **Sotuvlar**: какие колонки из дизайна обеспечены backend-API,
а какие — нет. Составлено по admin-спеке Swagger:
`http://46.225.154.145/swagger/admin/swagger.json` (тег **OrderAdmin**).

> TL;DR. Заказ в API — это **только «шапка»**. Полей **даты документа, номера
> документа, суммы и товаров** в контракте заказа нет вообще. Их нужно добавлять
> на бэкенде.

---

## 1. Эндпоинты заказа

REST под `/api/v1/admin/orders` (тег `OrderAdmin`):

| Метод | Путь | Назначение |
|-------|------|-----------|
| GET | `/admin/orders` | список (фильтры + пагинация) |
| POST | `/admin/orders` | создать |
| GET | `/admin/orders/{id}` | детальная карточка |
| PATCH | `/admin/orders/{id}` | обновить |
| DELETE | `/admin/orders/{id}` · `/admin/orders?ids=` | удалить (одиночно / массово) |

## 2. Что реально отдаёт API

**`OrderAdminList`** (строка списка) — ровно эти поля:
```
id, organization, client, status, paymentMethod, currency,
warehouse, delivery, rejectionReason, isDeleted
```
- `client`, `paymentMethod`, `currency`, `warehouse` приходят **строками** (отображаемые имена), не объектами.
- `status` — enum `OrderStatus`: `New | Accepted | Rejected | Progress | Completed | Delivered`.
- `delivery` — **boolean** (да/нет), НЕ статус доставки.

**`OrderAdminDetail`** (карточка по id) = поля списка + аудит:
```
created, updated, deleted (date-time), createdBy, updatedBy, deletedBy (пользователь)
```
> ⚠️ В Detail **нет** `organization` (важно для редактирования).

**`OrderAdminCreate`** (создание) требует:
```
organizationId*, warehouseId*, paymentMethodId*, currencyId*, status*, delivery*
clientId  (опционально)
```
Никаких товарных строк в создании заказа **нет**.

Проверено: во **всех** схемах Order (`Order`, `OrderAdminList/Detail/Create/Update`)
нет ни одного поля `date/number/total/sum/amount/price/document` — только аудит
`created/updated/deleted`.

## 4. Колонки БЕЗ API (нужен бэкенд) — 4 шт.

Сейчас на фронте это **плейсхолдеры** (рендерят `—` с тултипом):

1. **Hujjat sanasi** — дата документа. В `OrderAdminList` нет даже `created`.
   Просьба к бэку: добавить в список либо `created`, либо отдельное поле «дата документа».
2. **Hujjat raqami** — номер документа (на скрине «123»). Поля нет вообще.
   Просьба: добавить `documentNumber` (или аналог) в Order.
3. **Umumiy miqdor** — сумма заказа («5000$»). Поля нет вообще.
   Просьба: добавить вычисляемое `totalAmount` (+ валюта) в `OrderAdminList`.
4. **Tavar** — товарные строки заказа. В контракте заказа нет; продукция связана
   через `Plan` (у `Plan` нет своих admin-эндпоинтов).
   Просьба: отдать строки заказа (товар, кол-во, цена) — отдельным эндпоинтом
   `/admin/orders/{id}/items` или вложенным массивом в Order.

## 5. Колонки ЧАСТИЧНО (есть данные, но иначе) — 2 шт.

5. **Yetkazib berish** — `delivery` это **boolean**. Дизайн показывает статус-текст
   «Jarayonda» (в процессе). Если нужен статус доставки — на бэке нужно отдельное
   поле-enum, сейчас только да/нет.
6. **Bekor qilish** — причина отмены `rejectionReason` **есть в списке** ✅.
   А вот **дата отмены** (`deleted`) и **кто отменил** (`deletedBy`) — только в
   `OrderAdminDetail` (по одному заказу). Для поповера (i) можно дотянуть Detail —
   это реально по API уже сейчас.

## 6. Что уже сделано на фронте

Модуль `Sotuvlar` подключён к реальному API (`/sales`):
- **Реально работает:** список (статус-бейдж, склад, клиент, доставка, причина
  отмены), фильтры (статус → `Status`, склад → `WarehouseId`, клиент → `ClientId`,
  даты → `CreatedStart/End`), создание/редактирование шапки, удаление (вкл. массовое),
  выделение строк, настройка колонок.
- **Заглушки (`—`):** `Hujjat sanasi`, `Hujjat raqami`, `Umumiy miqdor`, `Tavar`.
- **Форма создания** — 6 селекторов (организация, клиент, склад, способ оплаты,
  валюта, статус) + переключатель доставки. Товарных строк нет (нет контракта).
- **Ограничение редактирования:** `OrderAdminDetail` не возвращает организацию,
  поэтому при правке её надо переселектить.
